# rebac-engine 기능 로드맵

이 문서는 **제어 엔진(rebac-engine + rebac-core + rebac-openfga-adapter)** 을, 귀하의 시스템에서 **권한 판단을 맡는 얕은 연결(shallow coupling) 계층** — 즉 **PDP(Policy Decision Point)에 가까운 코어** — 로 키우기 위한 기능·단계·빠뜨리기 쉬운 요소까지 한데 모은 것입니다. 구현 시 **Feature ID** 를 이슈·PR에 붙이면 추적이 쉽습니다.

---

## 이 엔진이 맡는 경계 (얕은 연결 PDP)

| 구분 | 엔진이 할 일 | 엔진이 하지 않는 일(상위 시스템·PEP 권장) |
|------|----------------|------------------------------------------|
| **결정** | “이 주체가 이 객체에 이 관계를 가질 수 있는가”와 그에 준하는 **집합·설명 질의**(ListObjects, Expand 등). | UI·API 게이트에서의 최종 차단만 두고 **비즈니스 규칙 전부**를 엔진에 몰아넣기. |
| **상태** | 관계 튜플의 **진실 공급원에 가깝게** 동기화·조회·삭제. | 도메인 엔티티 수명주기·트랜잭션 아웃박스 전부(필요 시 **이벤트 → 튜플** 은 별 파이프라인). |
| **결합** | OpenFGA(또는 다른 백엔드)와의 **프로토콜·재시도·한도**를 캡슐화. | 각 마이크로서비스의 **로컬 캐시 정책·세션** — 다만 엔진이 제공하는 **일관성·캐시 계약**은 문서로 맞춤. |
| **실패** | 백엔드 불가 시 **명시적 오류·degrade 정책**(아래 F-26). | 서비스별 **fail-open / fail-closed** 최종 선택(권한 민감도에 따라 다름). |

**“얕다”**는 의미: 비즈니스 서비스는 `user` / `relation` / `object` 문자열 규약과 엔진 API만 알면 되고, Zanzibar 전개·스토어·모델 버전은 엔진 뒤로 숨긴다.

## 진행 단계(Phase) — 재정렬

병렬 가능 항목은 같은 Phase에 둔다. **F-26** 은 Phase 1에서 문서만이라도 착수하는 것을 권장한다.

### Phase 0 — 기준선 (유지)

- 단일 `check` / `ingest`(`append`·`replace`, 빈 목록 시 스토어 비우기) / `list`.
- OpenFGA Read `pageSize` 등 **백엔드 제약과 어댑터 정합성** 회귀 테스트.

## SPI(`RebacEngine`) 설계 시 유의

- Phase 2 이후 메서드 증가 시 **F-31**: `TupleWriter`, `RelationshipReader`, `AccessChecker` 등 **역할별 인터페이스** 분리를 검토한다.
- OpenFGA가 아닌 백엔드를 허용할 경우 **F-02·F-03·F-06·F-25** 는 **지원 매트릭스**(가능/불가/제한)를 `docs/`에 유지한다.
- **F-23** 과 서비스 로컬 캐시가 동시에 있으면 **stale 허용 오류**가 중복된다. “엔진 캐시는 마지막 수단” 원칙을 문서에 쓴다.

## 다음 액션(이슈로 쪼개기 전)

1. Phase 1: **F-14 → F-13 → F-24 → F-15 → F-10 → F-26(문서) → F-17** 순으로 이슈 생성.  
2. 각 Feature에 **수용 기준** 한 줄(예: “모든 5xx에 `requestId` 포함”).  
3. **F-04** 스파이크: 기존 `GET /tuples` 호환 vs **신규 페이지 API** 브레이킹 여부 결정.  
4. **F-26** 과 보안 리뷰에서 **fail-open 금지** 여부를 제품 정책으로 확정.

이 파일은 Phase 안 순서만 조정하며 갱신하면 된다.
